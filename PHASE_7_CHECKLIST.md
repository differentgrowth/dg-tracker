# Phase 7 — Internal Report v1: Implementation Checklist

Temporary working file. Tracks the Phase 7 plan from `MVP_ROADMAP.md`. Delete on phase completion.

## Goal

Create an internal-only monthly report workflow that lets a signed-in team member generate a report for a client, persist the result, and read it inside the app.

## Locked decisions

- [x] **Report audience:** internal users only; no public sharing or client portal in MVP.
- [x] **Delivery:** in-app read-only report page first; PDF and email delivery are deferred.
- [x] **Period:** monthly reporting is the default, with explicit `periodStart` and `periodEnd` stored on `Report`.
- [x] **Data source:** report summaries are derived from stored `RankingSnapshot` and GSC performance snapshot data, not live GSC calls.
- [x] **Ranking semantics:** GSC `avgPosition` remains average position; do not present it as exact rank.

## 1. Schema & migration

- [ ] Decide whether to store the generated summary as JSON on `Report` or normalized report section rows.
- [ ] Add report summary fields needed for Phase 7, such as top wins, top losses, traffic deltas, keyword counts, and narrative notes.
- [ ] Add a report status if generation can fail or be retried (`draft`, `generated`, `failed`).
- [ ] Run `pnpm prisma migrate dev --name add_report_summary`.
- [ ] Run `pnpm prisma generate`.

## 2. Validators & actions

- [ ] Add `lib/validators/report.ts` for client ID and reporting period validation.
- [ ] Add a Server Action to generate a report for a client and period.
- [ ] Add a Server Action to refresh/regenerate an existing report if the user reruns the same period.
- [ ] Revalidate client report list and report detail pages after generation.

## 3. Report generation service

- [ ] Expand `lib/services/report.service.ts` with `generateClientReport(opts)`.
- [ ] Load the client, active keywords, ranking snapshots, and performance snapshots for the requested period.
- [ ] Compare against the previous equivalent period for deltas where data exists.
- [ ] Calculate top wins and top losses from average position changes.
- [ ] Calculate traffic deltas from clicks, impressions, CTR, and average position.
- [ ] Include clear empty states in the generated summary when insufficient comparison data exists.
- [ ] Persist the generated summary and return counts/IDs only; do not return raw GSC rows.

## 4. Reports UI

- [ ] Add a report generation form on the client reports surface.
- [ ] Add the read-only report detail route at `app/(dashboard)/clients/[clientId]/reports/[reportId]/page.tsx`.
- [ ] Render report sections for executive summary, KPI changes, top wins, top losses, and keyword opportunities.
- [ ] Link generated reports from the client page and reports list.
- [ ] Show report metadata: period, generated date, generated by, and data completeness.

## 5. Authorization & safety

- [ ] Require an authenticated session for all report pages and actions.
- [ ] Ensure users can only generate reports for existing non-archived clients.
- [ ] Avoid logging raw client data, GSC rows, or sensitive report contents.
- [ ] Surface safe, actionable errors at action boundaries.

## 6. Tests

- [ ] Add service tests for top wins/losses, traffic deltas, and no-comparison-data behavior.
- [ ] Add action validation tests for invalid client IDs and invalid date ranges if the existing test setup supports actions.
- [ ] Add route/component tests only if the UI logic becomes complex enough to justify them.
